创建联盟链

在启动节点之前，我们先确认项目已被完整编译完毕。

WASM_BUILD_TOOLCHAIN=nightly-2020-10-05 cargo build --release

在这一部分，我们将开始启动4个节点，包括3个 well known （知名）节点和1个普通节点。 这是本教程中会用到的node keys 和 peer ids。

Alice 的节点，

# node key
c12b6d18942f5ee8528c8e2baf4e147b5c5c18710926ea492d09cbd9f6c9f82a

# peerid
12D3KooWBmAwcd4PJNJvfV89HwE48nwkRmAgo8Vy3uQEyNNHBox2

# bs58 decoded peer id in hex:
0024080112201ce5f00ef6e89374afb625f1ae4c1546d31234e87e3c3f51a62b91dd6bfa57df

注意：你可以使用代码 bs58::decode 获取上述 bs58 解码格式的 peer id，类似于我们创世存储配置中的使用方式。 另外，为了方便，也可以使用一个简单的小工具来进行处理。

Bob 的节点，

# node key
6ce3be907dbcabf20a9a5a60a712b4256a54196000a8ed4050d352bc113f8c58

# peer id
12D3KooWQYV9dGMFoRzNStwpXztXaBUjtPqi6aU76ZgUriHhKust

# bs58 decoded peer id in hex:
002408011220dacde7714d8551f674b8bb4b54239383c76a2b286fa436e93b2b7eb226bf4de7

Charlie 的节点，

# node key
3a9d5b35b9fb4c42aafadeca046f6bf56107bd2579687f069b42646684b94d9e

# peer id
12D3KooWJvyP3VJYymTqG7eH4PM5rN4T2agk5cdNCfNymAqwqcvZ

# bs58 decoded peer id in hex:

Dave 的节点，

# node key 
a99331ff4f0e0a0434a6263da0a5823ea3afcfffe590c9f3014e6cf620f2b19a

# peer id
12D3KooWPHWFrfaJzxPnqnAYAoRUyAHHKqACmEycGTVmeVhQYuZN

# bs58 decoded peer id in hex:

Alice 和 Bob 的节点已经配置在创世文件中，并且作为 well known 节点。 之后，我们将 Charlie 的节点添加到 well known 节点集合中。 最后，我们可以把 Charlie 的节点和 Dave 的节点连接起来，而不用把 Dave 的节点也作为 well known 节点。

启动 Alice 和 Bob 的节点

我们先启动 Alice 的节点：

./target/release/node-template --chain=local --base-path /tmp/validator1 --alice --node-key=c12b6d18942f5ee8528c8e2baf4e147b5c5c18710926ea492d09cbd9f6c9f82a --port 30333 --ws-port 9944

我们在这里使用 --node-key 来指定用于网络安全连接的密钥。 这个密钥也用于生成具有可读性的 PeerId ，如上节所示。

其他用到的 CLI flag 如下：

  • --chain 用于本地测试网。
  • --alice 使节点成为一个可以生产和最终确定区块的权威节点，同时给节点取名为 alice。
  • --port 为p2p连接指定一个端口。
  • --ws-port 为WebSocket 连接指定一个监听端口。

您可以通过运行 ./target/release/node-template -h 来获取以上flags的详细描述。

启动 Bob 的节点。

./target/release/node-template --chain=local --base-path /tmp/validator2 --bob --node-key=6ce3be907dbcabf20a9a5a60a712b4256a54196000a8ed4050d352bc113f8c58 --port 30334 --ws-port 9945

在这两个节点启动后，你应该能够看到新的区块被生产出来并最终确认。 现在让我们使用 polkadot.js apps 来检查我们的区块链中的 well known 节点。 别忘了使用我们的本地节点 127.0.0.1:9944。

首先，我们需要增加一个额外的设置来告诉前端在node-authorization pallet中使用的 PeerId的类型。 注意：这里 PeerId 是对 bs58 解码后格式，以字节集合的形式封装在 PeerId 结构体里。 进入 app 的 Settings Developer 页面，添加以下类型信息：

{
  "PeerId": "(Vec<u8>)"
}

然后，我们进入 Developer Chain state 页面，查看 nodeAuthorization 模块、wellKnownNodes 存储的数据。 你应该可以看到 Alice 和 Bob 节点的 peer Id，前缀为0x，表示以十六进制格式显示其字节集合。

我们还可以通过查询存储 owners，以节点的 peer id 为输入，查询一个节点的所有者，应该可以得到所有者的账号地址。

添加另一个知名节点

让我们启动 Charlie 的节点，

./target/release/node-template --chain=local --base-path /tmp/validator3 --name charlie  --node-key=3a9d5b35b9fb4c42aafadeca046f6bf56107bd2579687f069b42646684b94d9e --port 30335 --ws-port=9946 --offchain-worker always

启动后，您应该看到此节点没有相连的节点。 这是因为我们已经在一个许可网络中，您需要获得授权才能连接！

刚提到过，我们使用 sudo 模块进行治理，我们可以使用 sudo 权限调用 node authorization pallet 提供的 add_well_known_node，来添加我们的节点。 您可以在这个 API 文档 中找到更多的调用命令。

在 apps 应用中跳转到 Developer Sudo 页面，提交 nodeAuthorization - add_well_known_node 交易，参数为用十六进制表示的 Charlie 节点的 peer id 和所有者也就是 Charlie。

当交易打包入区块中后， 你应该能看到 Charlie 的节点已连接到 Alice 和 Bob 的节点，并逐渐进行区块同步。 在本地网络中默认启用了 mdns 发现机制，因此三个节点可以互相找到。 如果在公共互联网中运行节点，您需要在启动 Charlie 的节点时使用 --reserved-nodes flag 来指定可访问的节点。 或将可访问的节点作为 Chain Spec 中的启动节点。

现在我们已经有 3 个 well known 节点了。

添加一个额外的连接

现在让我们增加另外一个节点，不过它不是 well known 节点，而是 Charlie 节点的“子节点”。

使用以下命令启动节点。

./target/release/node-template --chain=local --base-path /tmp/validator4 --name dave --node-key=a99331ff4f0e0a0434a6263da0a5823ea3afcfffe590c9f3014e6cf620f2b19a --port 30336 --ws-port 9947 --offchain-worker always

在启动后，没有任何节点间的可用连接。 首先，Charlie 需要配置他的 well known 节点，允许 Dave 的节点与其连接。

在 Developer Extrinsics 页面中，Charlie 提交了一个 addConnections 交易。 第一个 PeerId 是 Charlie 节点中的 peer id。 connections 这一参数是允许连接到 Charlie 节点的节点集合，在此我们只添加 Dave 的节点。

然后，Dave 需要配置他的节点来允许来自 Charlie 节点的连接。 但在添加连接之前，Dave 需要先认领他的节点。希望这不会太晚！

同样，Dave 可以添加来自 Charlie 节点的连接。

若没有连接到其他节点，则请重启 Dave 的节点。 你现在应该看到 Dave 正在同步区块，并且只连接了一个节点，是属于 Charlie 的。

恭喜你！

你已经完成了本教程，并了解了如何建立一个新的需要授权的区块链网络。 你也可以测试其他可调用函数，如 remove_well_known_node, remove_connection。 请随时与我们联系并分享你在构建许可区块链中所学习到的的经验。
